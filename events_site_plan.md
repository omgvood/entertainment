# Events Site — план MVP

> Веб-сайт-агрегатор досуга в формате маркетплейса (WB/Ozon-стиль).
> Старт: Пермь. Архитектурно — мультигород.
> Зафиксировано: 2026-05-22. Предыдущая версия (Telegram-бот) переписана под сайт.

## 1. Контекст и позиционирование

**Проблема:** житель города хочет куда-то сходить в свободное время, но не знает куда. Нужен инструмент «выбрать развлечение из списка как с маркетплейса».

**Формат:** веб-сайт с фильтрами и сеткой карточек (4 в ряд десктоп, 2 на мобиле), переключатель города в шапке.

**Стратегия ниши:**
- **MVP-1:** узкая ниша — то, что плохо покрывает Я.Афиша: квизы, стендапы, боулинг, бильярд, картинг. Реалистичные шансы выйти в топ SEO по «квизы Пермь», «куда сходить вечером Пермь».
- **MVP-2+:** расширение в широкую афишу (концерты, кино, театр, выставки), когда есть SEO-база и первый трафик.

**Модель:** pet-project, без монетизации. Метрики успеха: индексация в Яндексе, позиции по ключам, посетители (Я.Метрика).

## 2. Архитектура

**Идея:** статический сайт + ночная пересборка. Естественно ложится на требование «обновление каждый день в 00:00».

```
[Парсеры Python] → [Postgres] → [Сборка статики ночью] → [CDN] ← [Пользователь]
       ↑                              ↑
       └── cron GitHub Actions ───────┘
```

**Почему статика, а не SSR:**
- SEO: статика индексируется идеально, без подводных камней гидратации.
- Скорость: без сервера → быстрее → лучше для SEO и мобильного UX.
- Стоимость: 0 ₽ на Vercel/Cloudflare Pages.
- Обновление раз в сутки уже описано как ночное → SSR избыточен.

**Триггер для смены подхода:** если событий станет >10k или фильтрация в браузере просядет — миграция на ISR в том же Next.js.

## 3. URL-структура и SEO

| URL | Назначение | SEO-цель |
|---|---|---|
| `/` | 301-редирект на `/perm/` (дефолтный город) | — |
| `/perm/` | главная Перми | «куда сходить в Перми», «афиша Пермь» |
| `/perm/today/` | события сегодня | «куда сходить сегодня Пермь» |
| `/perm/weekend/` | события на выходные | «куда сходить на выходных Пермь» |
| `/perm/quiz/` | срез по типу | «квизы Пермь» |
| `/perm/standup/` | срез по типу | «стендап Пермь» |
| `/perm/events/<slug>/` | страница события | длинный хвост: «квиз мозгобойня соль Пермь» |
| `/ekb/...` | другой город | масштабирование |
| `/sitemap.xml`, `/robots.txt` | для краулеров | — |

**Чек-лист SEO MVP-1:**
- SSG (статическая генерация всех страниц).
- meta title/description по шаблону на каждую страницу.
- Open Graph (для шеринга в Telegram/VK).
- Schema.org `Event` JSON-LD на странице события.
- sitemap.xml автогенерируется при сборке.
- Канонические URL.
- Регистрация в Я.Вебмастере + Google Search Console + Я.Метрике.
- Альты у картинок: `{title} — {type} в {venue_name}`.

## 4. UI/UX

Референс: WB/Ozon (скрины — [image_2026-05-22_09-41-35.png](image_2026-05-22_09-41-35.png), [photo_2026-05-22_09-41-35.jpg](photo_2026-05-22_09-41-35.jpg)).

**Главная города:**
- Шапка: лого / селектор города / поиск / пиксель Я.Метрики.
- Левая колонка (десктоп) или drawer-фильтр (моб): тип, дата (сегодня/завтра/выбрать), цена (слайдер), район — поле в БД готовим сразу, фильтр в v2.
- Сетка карточек: **4 в ряд (≥1024px), 3 (≥768), 2 (<768)**.
- Карточка: фото / тип / название / дата + время / цена «от X ₽» / адрес.
- Пагинация (для SEO предпочтительнее бесконечной прокрутки).

**Чекбокс «только события с фиксированной датой»** — включён по умолчанию, чтобы боулинг не лез в выдачу при поиске «суббота вечер».

**Страница события:** полная карточка + описание + организатор + кнопка «Перейти к источнику».

## 6. Источники и парсинг

**Принцип:** масштабирование на 100 городов исключает написание парсера под каждый сайт. Архитектура — двухуровневая, без ручного ввода данных.

### Дискавери (код, бесплатно)
Краулер берёт seed-список источников на каждый город (агрегаторы, площадки, тематические ВК-сообщества, Telegram-каналы) и собирает ссылки на события через универсальные паттерны: RSS, sitemap.xml, JSON-LD внутри HTML, ссылки в листингах. **Один Python-код работает для всех источников.** Seed-список — это конфиг (~10-30 URL на город), вносится один раз при подключении города.

### Извлечение (LLM)
Для каждой найденной URL — запрос к дешёвой LLM (Claude Haiku 4 / Llama-3.3 через OpenRouter) с фиксированной JSON-схемой события. **Один промпт работает на любом сайте** — парсеры под конкретные сайты не нужны. Дальше валидация по схеме (дата парсится, цена ≥ 0, обязательные поля заполнены) → запись в БД.

Оценка стоимости: ~$0.001-0.005 за событие.
- MVP-1 (1 город, ~50 событий/день): <$10/мес.
- Потолок (100 городов × 50 событий × 30 дней): $150-700/мес. Дедупликация по URL и хешу контента срезает 80-90% — LLM зовётся только на новых/изменившихся страницах.

### Места «всегда» (боулинг, бильярд, картинг)
Парсинг 2ГИС / Я.Карт по категории и городу — один код для всех городов. Никакого ручного ввода CSV.

### Регламент обновления (GitHub Actions cron, 00:00 МСК ежедневно):
1. Дискавери-краулер по всем городам → список URL-кандидатов.
2. Дедупликация против БД — LLM зовётся только для новых/изменившихся URL.
3. LLM-извлечение → валидация по JSON-схеме → запись в Postgres.
4. Health-check: если поток новых событий < исторического минимума (например, <50% от среднего) — алерт в личный Telegram.
5. Триггер сборки сайта (Next.js build) → деплой на Vercel/Cloudflare Pages.

### Что НЕ делаем (и почему)
- **Автономный LLM-агент, который «сам лазит» по интернету** — слишком медленно/дорого/ненадёжно для ежедневного прогона. Используем LLM только как извлекатель структуры из конкретной страницы.
- **Парсеры под отдельные сайты** (QuizPlease, Мозгобойня и т. п.) — не масштабируется на 100 городов, ломается при редизайнах источников.

## 7. План спринтов

1. **Спринт 0 (1 вечер):** Next.js + хардкод 10 событий Перми → деплой на Vercel. Цель: увидеть UX в браузере, отладить сетку.
2. **Спринт 1 (1-2 вечера):** Postgres (Supabase free) + ручная заливка 20-30 реальных событий + Next.js читает из БД при build → SSG. Цель: статика собирается из БД.
3. **Спринт 2 (1 вечер):** SEO-минимум — meta, sitemap, schema.org Event. Регистрация в Я.Вебмастере. Цель: подать на индексацию.
4. **Спринт 3 (2-3 вечера):** дискавери-краулер для 5 seed-источников Перми + LLM-извлечение через единый промпт + GitHub Actions cron. Цель: автообновление через универсальный пайплайн (не парсер на каждый сайт).
5. **Спринт 4 (1-2 вечера):** парсер 2ГИС по категориям «боулинг / бильярд / картинг» для Перми. Цель: «места всегда» масштабируются на любой город одним кодом.
6. **Спринт 5:** Я.Метрика + Google Search Console + валидация индексации. Цель: видеть трафик и позиции.

После MVP-1: расширение типов (концерты/кино/театр), второй город, гео-фильтр по району.

## 8. Технологический стек

- **Frontend:** Next.js 14+ (App Router, SSG), TypeScript, Tailwind CSS.
  - *Альтернатива:* Astro — проще и быстрее для контентного сайта, но меньше экосистемы под богатые фильтры. Если плотных фильтров не планируется — взвесить.
- **БД:** PostgreSQL (Supabase free / Neon free).
- **Парсинг:** Python 3.11+, `httpx`, `selectolax` для дискавери; при необходимости `Playwright` для JS-сайтов.
- **LLM-извлечение:** Claude Haiku 4 (через Anthropic API) либо Llama-3.3 70B / Qwen-2.5 (через OpenRouter) — извлечение структурированного события из HTML.
- **Шедулер:** GitHub Actions cron (бесплатно).
- **Хостинг:** Vercel или Cloudflare Pages (free).
- **Аналитика:** Яндекс.Метрика.
- **Хранение картинок:** Supabase Storage / Cloudflare R2.

## 9. Риски и решения

| Риск | Влияние | Решение |
|---|---|---|
| SEO-конкуренция с Я.Афишей | Трафик | Узкая ниша на старте: «квизы Пермь», «стендап Пермь» — там Афиша слаба |
| Места «всегда» загрязняют выдачу по дате | UX | Чекбокс «только события с датой» включён по умолчанию |
| Парсеры ломаются при редизайне источников | Качество данных | LLM-извлечение адаптивно к разметке; Health-check + алерт в Telegram при просадке потока |
| LLM галлюцинирует поля (даты, цены) | Качество данных | Строгая JSON-схема + валидация (даты парсятся в datetime, цена ≥ 0, обязательные поля); невалидные события отбрасываются |
| Стоимость LLM растёт с числом городов | Бюджет | Дедупликация по URL и hash контента — LLM зовётся только на новых страницах. Откат на open-source модели через OpenRouter если Haiku дорогой |
| Картинки источника защищены/удаляются | UX | Кешировать в Storage при парсинге, fallback-плейсхолдер по `type` |
| Юридика парсинга / ToS площадок | Низкий для пет-проекта | Обратная ссылка `source_url` на каждой карточке |
| Build time растёт с числом событий | Скорость деплоя | Перейти на ISR, когда build > 5 мин |

## 10. Что отложено на v2

- Гео-фильтр по району/метро (`district` собираем сразу).
- Полная геолокация «ближайшее ко мне».
- Второй город — после стабилизации Перми.
- Подписки на категории + email/push о новых событиях.
- Расширение в широкую афишу (концерты/кино/театр) — после первых SEO-результатов в нише.
- Telegram-бот как второй канал поверх той же БД.
- Монетизация.
